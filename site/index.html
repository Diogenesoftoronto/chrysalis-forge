Core Architecture

THE
EVOLUTION
ENGINE

Chrysalis doesn't just run tasks — it runs a continuous improvement loop. After each session, it mutates candidate prompts and strategies, evaluates them against your real workload, and archives the winners. The losers get thrown into the evolutionary scrap heap.

This is MAP-Elites: a quality-diversity algorithm that maintains a library of diverse high-performing candidates across behavioral niches — latency, cost, token efficiency. When you set a priority, you're selecting which niche to pull from.

"Reflective prompt evolution outperforming RL-based methods on downstream tasks." — GEPA paper (arXiv:2507.19457)
01

Novelty Detection

Jaccard n-gram similarity rejects mutations too close to existing elites. No wasted LLM calls on trivial variants.

02

Bandit Model Ensemble

Thompson Sampling (Beta-Binomial conjugate prior) picks which model generates mutations — the ones that win stay in rotation.

03

Cross-Model Generalization

Evolved configs are validated across held-out models. If it only works for one model, it doesn't make the archive.

04

Harness Strategy Evolution

12 evolvable fields: context budget, temperature, tool routing, execution priority, demo selection. The harness itself evolves.

Operational Modes

FOUR MODES.
ONE SWITCH.

Modes are role-based access control for tools. Switch mid-session with /mode <name>. Think of them as loadout profiles — each locks in the right surface area for the job.

ASK
/mode ask

Conversational only. No filesystem, no network. Safe for experimentation and ideation. The "observer" mode.

ARCHITECT
/mode architect

Read-only file access. Perfect for code review, analysis, and getting oriented in an unfamiliar codebase.

CODE
/mode code

Full capabilities. File writes, shell commands, network access, sub-agents. This is where the work gets done.

SEMANTIC
/mode semantic

RDF knowledge graph operations and vector memory. For agents that reason over structured knowledge bases.

Permission Model

CLEARANCE
LEVELS

Two orthogonal axes: modes control which tools are available, levels control how dangerous operations are handled. Set with --perms <level>.

0
SANDBOX
  • Conversation only
  • Sandboxed Racket eval
  • No filesystem
  • No network
1
ANALYST
  • Filesystem read
  • Network read
  • Sandboxed Racket eval
  • No writes
2
ENGINEER
  • Filesystem write
  • Requires confirmation
  • Optional LLM judge
  • No shell
3
OPERATOR
  • Full shell access
  • Requires confirmation
  • LLM judge available
  • Full filesystem
∞
GOD
  • Auto-approve all
  • No confirmations
  • Full capabilities
  • Use carefully

* An optional LLM security judge (LLM_JUDGE=true) reviews operations before execution at levels 2–3, responding [SAFE] or [UNSAFE]. It doesn't catch everything, but it catches the obvious.

Adaptive Execution

NATURAL LANGUAGE
PRIORITY
SELECTION

You don't have to know which model configuration is best for your task. Describe what you need — in keywords or plain English — and Chrysalis uses KNN search in a normalized phenotype space to find the archived agent variant that best matches your stated priorities.

The agent can also switch its own priority mid-task using the set_priority tool. Hit a compute-intensive subtask? It upgrades to "best." Hit a routine one? Drops to "cheap." Zero manual intervention.

$ chrysalis --priority fast "Summarize this file"
Low latency. Accepts slightly lower accuracy. KNN maps to the speed-optimized niche.
$ chrysalis --priority cheap "Batch analyze 50 files"
Minimize token cost. Selects cheapest model variant with acceptable quality.
$ chrysalis --priority "I need accuracy but I'm on a budget" "Review security code"
LLM interprets the natural language into phenotype coordinates. Balanced niche selected.
$ chrysalis --priority "I'm in a hurry but this is critical" "Deploy script review"
Tension detected. System finds the Pareto frontier between speed and accuracy in the archive.
$ chrysalis --priority best "Deep analysis of codebase"
Highest quality regardless of cost or latency. The thoroughbred variant.

Get Started

READY TO
INSTALL

Requires Racket v9.0+. Three steps and you're running.

1

Clone the repository

git clone https://github.com/Diogenesoftoronto/chrysalis-forge.git
cd chrysalis-forge
2

Install package dependencies

raco pkg install --auto

This registers chrysalis and chrysalis-client as commands.

3

Configure your API key

echo "OPENAI_API_KEY=sk-..." > .env

Works with any OpenAI-compatible API — LiteLLM, Ollama, vLLM, etc.

→

Launch

chrysalis -i

Interactive mode. Start talking.

TIP — For local models, set OPENAI_API_BASE=http://localhost:11434/v1 and MODEL=llama3.2 to use Ollama. Any OpenAI-compatible endpoint works.

ENVIRONMENT VARIABLES

OPENAI_API_KEY
REQUIRED
Your API key. Works with any OpenAI-compatible provider.
OPENAI_API_BASE
optional
Custom API endpoint. For LiteLLM, Ollama, vLLM, or any compatible API.
MODEL
optional
Override the default model (e.g. gpt-4o, llama3.2).
EXA_API_KEY
optional
Exa AI for neural web search. Falls back to DuckDuckGo via curl without it.
LLM_JUDGE
optional
Set to true to enable LLM-based security review of operations.
BUDGET
optional
Session spend cap in USD (e.g. 2.50). Stops gracefully when reached.
TIMEOUT
optional
Session time limit (e.g. 10m, 1h).
CHRYSALIS_SECRET_KEY
optional
JWT secret for service mode (--serve) with authentication.

Who it's for

FIND YOUR
MISSION PROFILE

Chrysalis isn't for everyone. It's for the people who looked at existing AI tools and thought "I want to go deeper."

THE RESEARCHER
arXiv reader · agent systems · probably has strong opinions about eval
You're studying agent architectures. You want to run controlled experiments with MAP-Elites on real tasks, compare GEPA vs naive prompting, instrument the evolution loop, and read the traces. The eval store and lineage tracking are built for you.
chrysalis --priority accurate "Evaluate three prompt variants on this test suite"
chrysalis -i
/stats
THE BUILDER
solo dev · moving fast · tired of babysitting agents
You ship things. You want an agent that reads your project rules, follows your commit conventions, makes surgical edits (not full rewrites), and can spawn parallel sub-agents when tasks get complex. You want transparency, not magic.
chrysalis --perms 2 "Refactor the parser module — make it testable"
chrysalis --perms 2 "Write tests for everything in src/core/"
THE OPERATOR
infra / automation · multi-user deployment · API integrations
You need Chrysalis running as a service. Multiple users, JWT auth, rate limiting, billing tracking, OpenAI-compatible API endpoint. Point your existing tools at it and get an evolution-capable backend.
chrysalis --serve --serve-port 8080 --daemonize
chrysalis-client --url http://your-server:8080 --api-key <token>

THE TINKERER

Local models, custom endpoints, Ollama, vLLM, LiteLLM — Chrysalis speaks standard OpenAI protocol so it works with anything. Set OPENAI_API_BASE and MODEL and you're running Llama locally with full evolution capabilities.

THE DEBUGGER

Debug mode reveals which tools models actually use vs ignore. The traces are full JSONL — machine-readable telemetry for every tool call, token count, cost, and latency. ~/.chrysalis/traces.jsonl is your ground truth.

Theoretical Foundations

BUILT ON
REAL SCIENCE

Not vibes. Not "inspired by." Direct implementations of peer-reviewed research.

GEPA: General Evolvable Prompting Architecture
arXiv:2507.19457
Reflective prompt evolution that outperforms RL-based methods on downstream tasks. The core of Chrysalis's /evolve command and evolve_system tool.
MAP-Elites: Illuminating Search Spaces through Quality Diversity
arXiv:1504.04909
Quality-diversity optimization maintains a library of diverse high-performing candidates. Chrysalis uses it to maintain an archive of agent variants across latency/cost/accuracy niches.
MAKER: Million-Step Zero-Error Reasoning via Extreme Decomposition
arXiv:2511.09030
Geometric task decomposition for zero-error reasoning at scale. Informs Chrysalis's geometric decomposition system in src/core/geometric-decomposition.rkt.
Graphiti/Zep: Temporal Knowledge Graphs for Agent Memory
arXiv:2501.13956
Temporal knowledge graphs enabling agents to reason over time-structured information. Informs Chrysalis's RDF semantic memory and vector store architecture.
Grassmann Flows: Geometric Alternatives to Attention
arXiv:2512.19428
Geometric alternatives to transformer attention mechanisms. Informs phenotype space design and the KNN search used in natural language priority selection.
Recursive LMs: Unbounded Context via Recursive Decomposition
arXiv:2512.24601
Recursive decomposition enables unbounded context handling. Motivates Chrysalis's context compaction strategy and the multi-tier context management system.
